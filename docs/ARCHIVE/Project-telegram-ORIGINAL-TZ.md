# Проект: Telegram‑бот для раздачи холодных лидов с интеграцией в Bitrix24

## Контекст

Ты — опытный системный архитектор и backend‑разработчик (Python/Telegram‑боты + Bitrix24). Твоя задача — выдать детальное, но практически применимое архитектурное описание системы (без продакшн‑кода), по которому можно писать ТЗ и начинать разработку.

Исходные условия:

- Есть внешняя программа, которая из одного источника собирает холодные лиды (телефоны компаний) по разным сегментам, очищает их и вручную формирует CSV‑файлы с разделителем запятая.  
- CSV содержит поля, сопоставимые с сущностью «Лид» в Bitrix24 (минимум: название компании, телефон, адрес, город, сегмент и т.п.)
- CSV импортируются в базу SQLite на локальном сервере. Активный объём — до ~10 000 лидов.  
- После импорта каждого нового лида в SQLite выполняется автоматическая проверка на дубль в Bitrix24 по телефону, названию компании и адресу.  
  - Используются REST‑методы поиска по коммуникациям и дублям (например, crm.duplicate.findbycomm, обзор по работе с дублями и полями для поиска дублей ).  
  - Лиды, для которых найден дубль, помечаются статусом `DUPLICATE` и никогда не выдаются менеджерам.  
  - Лиды без дублей помечаются как `UNIQUE`/актуальные и попадают в пул доступных для выдачи.  
- Примерно 20 менеджеров работают через Telegram‑бот (интерфейс только на русском языке). Все лиды — холодные, выдаются по единым правилам.  
- Менеджер может за один запрос получить не более 200 лидов.  
- Аналитика должна давать базовые метрики по дням/неделям/месяцам:  
  - сколько лидов загружено;  
  - сколько отмечено дублями и процент дублей;  
  - сколько выдано и импортировано по сегментам, городам и менеджерам.  
- Админ (владелец системы) через Telegram‑бот:  
  - импортирует новые CSV в базу SQLite (через функцию бота);  
  - запускает автоматическую и повторную проверку дублей;  
  - просматривает агрегированную статистику;  
  - экспортирует CSV‑отчёты;  
  - запускает ежемесячную очистку логов/аналитики;  
  - запускает очистку старых лидов со статусами `DUPLICATE` и `IMPORTED` (раз в месяц и вручную);  
  - может временно «заморозить» определённые сегменты (или сегменты в конкретных городах), чтобы лиды этих сегментов не выдавались менеджерам и не попадали в динамические кнопки, при этом физически оставаясь в БД.  
- В Telegram‑боте используются динамические кнопки (inline/reply‑клавиатуры), формируемые **только по доступным данным**:  
  - список сегментов и городов всегда отражает только те сегменты/города, где есть доступные и не замороженные лиды (статус `UNIQUE` и не помечены как «замороженные»);  
  - если для сегмента или города доступных лидов нет, либо сегмент «заморожен», соответствующая кнопка **не отображается** .  
- При запросе менеджером N лидов (N ≤ 200), если в выбранном сегменте/городе доступно меньше лидов (например, 10 вместо запрошенных 20), бот должен:  
  - сообщить, что выдать N лидов нельзя;  
  - предложить выдать доступное количество (10);  
  - запросить подтверждение;  
  - по подтверждению выдать доступное количество лидов.  

Структура файлов и референсы:

- На сервере предусмотрена папка `/uploads`, в которой хранятся готовые CSV‑файлы с лидами.  
  - Эти файлы используются как исходные данные для импорта в базу SQLite.  
  - Также файлы в `/uploads` служат референсом для разработчика: по ним можно понять фактическую структуру полей, формат маппинга под Bitrix24 и возможные вариации данных.  
  - В архитектурном описании укажи, что модуль импорта по умолчанию ищет CSV именно в `/uploads` (если админ не отправляет файл непосредственно в бота), а также что админ через бота может запускать импорт новых файлов из этой папки.

Отдельное важное требование: процесс регистрации менеджеров и привязки к Bitrix24.

- При первом запуске бота менеджером:  
  - бот не даёт сразу получать лиды, а запускает процедуру регистрации;  
  - бот просит написать Имя и Фамилию (свободным текстом), чтобы админ мог идентифицировать менеджера в Bitrix24;  
  - после этого создаётся заявка, которая отправляется админу в боте (приглашение «привязать» Telegram‑аккаунт к конкретному пользователю Bitrix24);  
  - админ, получив заявку, выбирает или вводит ID пользователя Bitrix24 и подтверждает привязку;  
  - после подтверждения админом Telegram‑аккаунт менеджера считается «активированным» и привязанным к конкретному ID в Bitrix24;  
  - только после этого менеджер может полноценно пользоваться функцией получения лидов и загрузки в Bitrix24 по API.  

Стек можно рекомендовать как вариант (например, Python + aiogram 3.x для бота, SQLite для БД, Bitrix24 REST API для работы с лидами и дублями), но не обязательно привязываться к конкретным библиотекам.

Не пиши реальный код. Используй архитектурные описания, текстовые схемы, псевдокод и примеры сообщений/кнопок.

Дополнительные условия:

- Система разворачивается и используется в РФ. На данный момент у Telegram и Telegram API периодически возникают проблемы со стабильностью подключения и доступностью с территории РФ (блокировки, нестабильные каналы и т.д.). Это нужно учитывать в архитектуре: предусмотреть корректную обработку временной недоступности Telegram, возможность перезапуска и восстановления состояния, а также, при необходимости, варианты использования прокси/VPN или промежуточных прокси‑сервисов (на уровне инфраструктуры, а не логики бота).  
- Интеграция с Bitrix24 должна осуществляться через **входящий вебхук** с минимально необходимым набором прав, в первую очередь на модули CRM и Users (создание и чтение лидов, поиск дублей, работа с пользователями). В архитектуре явно опиши, какие действия выполняются через этот вебхук и какие права для них нужны.  
- При проектировании решения предполагается использование современных инструментов оркестрации и расширения возможностей модели: **агенты, mcp‑серверы, скиллы, context7, доступ к свежей документации и интернет/браузеру при необходимости**. То есть при уточнении деталей API Bitrix24, Telegram Bot API, особенностей работы в РФ и т.п. модель должна опираться на актуальную документацию и при необходимости обращаться к внешним источникам через браузер/инструменты, а не полагаться на устаревшие знания.

Важно: при проектировании решения ты должен **всегда**:

- Использовать доступные инструменты: агентов, mcp‑серверы, скиллы, механизмы вроде context7 для работы с контекстом.  
- При необходимости обращаться к **свежей документации** (Bitrix24 REST API, Telegram Bot API, библиотеки для ботов, особенности работы в РФ и т.д.) через интернет и браузер‑инструменты, а не полагаться на устаревшие знания.  
- Проявлять осторожность с предположениями: если какая‑то деталь (например, конкретный метод API или ограничения вебхука) может меняться, сначала проверяй это по актуальной документации.

## 2. Структура SQLite и статусы лидов

Опиши логическую схему БД:

- Таблица `leads`:  
  - поля:  
    - `id` (внутренний ID);  
    - данные из CSV: телефон, название компании, адрес, город, сегмент/направление, источник;  
    - статусы:  
      - `NEW` — только что импортирован, ещё не проверен на дубль;  
      - `UNIQUE`/`ACTIVE` — проверен, дублей не найдено, доступен для выдачи (если сегмент/город не заморожены);  
      - `DUPLICATE` — найден дубль в Bitrix24, не выдаётся;  
      - `ASSIGNED` — лид выдан менеджеру (в процессе импорта или перед ним);  
      - `IMPORTED` — лид успешно создан в Bitrix24;  
      - при необходимости — статусы ошибок (`ERROR_IMPORT`).  
    - связи: id менеджера (если выдан/импортирован), id лида в Bitrix24;  
    - временные метки: дата/время создания, проверки дублей, выдачи, импорта.  
- Таблица `users`:  
  - Telegram‑ID;  
  - ФИО, указанные менеджером при регистрации;  
  - роль (`manager`/`admin`);  
  - привязка к пользователю Bitrix24 (ID), заполняется после подтверждения админом;  
  - статус (`PENDING_APPROVAL`, `ACTIVE`, при необходимости `REJECTED`);  
  - дата регистрации, дата подтверждения.  
- Таблица `segment_locks` (или иной механизм «заморозки»):  
  - сегмент;  
  - опционально город (если заморозка возможна по сегменту в конкретном городе);  
  - флаг `is_frozen` (true/false);  
  - дата установки/снятия заморозки;  
  - комментарий администратора (опционально).  
- Таблица `logs`:  
  - тип события (импорт CSV, проверка дублей, выдача лидов, импорт в Bitrix24, ошибка, запрос статистики, очистка логов, очистка лидов, заморозка/разморозка сегмента и т.п.);  
  - пользователь (если есть);  
  - связанные id лидов/сегментов/городов;  
  - дата/время;  
  - текстовое описание.  
- Подход к аналитике:  
  - агрегированные таблицы (`daily_stats`, `weekly_stats`, `monthly_stats`) или расчёт по `leads` и `logs`;  
  - метрики: загружено, дублей, процент дублей, выдано, импортировано по сегментам/городам/менеджерам для дней/недель/месяцев.  
- Таблица/механизм для отметки последней очистки и удаления старых записей, включая лиды `DUPLICATE` и `IMPORTED`.

Особенности для РФ и Bitrix24:

- При развёртывании в РФ важно учитывать возможные ограничения/блокировки Telegram:

  - При необходимости настроить прокси/VPN или использовать инфраструктуру вне РФ, а сам бот/бекенд связывать с локальной сетью (это решается на уровне DevOps/администратора).  
  - В инструкциях по эксплуатации указать, что при длительной недоступности Telegram менеджеры не смогут работать с ботом, но база лидов и интеграция с Bitrix24 должны оставаться консистентными.

- Для Bitrix24 использовать входящий вебхук:

  - В административной части Bitrix24 создать входящий вебхук с правами на CRM и Users (как минимум).  
  - Сохранить URL вебхука и ключ в конфигурационный файл проекта.  
  - В конфигурации явно выделить блок с параметрами вебхука (например: `BITRIX_WEBHOOK_URL`, `BITRIX_WEBHOOK_SECRET` или аналогично), и в архитектурном описании указать, что все обращения к Bitrix24 идут через этот вебхук.
