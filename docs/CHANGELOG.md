#### Загрузка лидов на Bitrix24 ID (незарегистрированные менеджеры)

**Файлы:**
- `src/bot/handlers/admin_load_leads.py` — расширен для поддержки Bitrix24 ID

**Возможности:**
- Админ может загрузить лиды на любого пользователя Bitrix24 по ID
- Не требуется регистрация менеджера в боте
- Лиды создаются в Bitrix24 с привязкой к ответственному
- Лиды не привязываются к менеджеру в боте (manager_telegram_id=NULL)

**Сценарий:**
1. Админ нажимает "📤 Загрузить на Bitrix24 ID"
2. Вводит Bitrix24 ID пользователя
3. Выбирает сегмент и город
4. Вводит количество лидов
5. Подтверждает загрузку
6. Лиды импортируются в Bitrix24

**Миграция:** Не требуется (используется существующая схема)

### 🐛 Исправления

#### Исправления багов админ-панели

**Проблема 1: Сегменты не показываются при загрузке лидов**

**Причина:** Функция `get_segments_with_cities()` возвращала только сегменты с городами, обработчик ожидал название сегмента, а получал индекс.

**Решение:**
- `src/database/crud.py` — возвращает все сегменты, даже без городов
- `src/bot/handlers/admin_load_leads.py` — обработка индекса вместо названия

**Проблема 2: Очистка данных не работает**

**Причина:** Неправильный порядок роутеров — общий роутер `cleanup_` перехватывал callback `cleanup_confirm:logs`.

**Решение:**
- `src/bot/handlers/admin_handlers.py` — поменян порядок роутеров (специфичный перед общим)
- `src/cleanup/cleanup_service.py` — добавлен `flush()` перед `commit()`

**Foreign Keys для SQLite**

**Файл:** `src/database/models.py`

**Добавлено:**
```python
@event.listens_for(self.engine.sync_engine, "connect")
def set_sqlite_pragma(dbapi_connection, connection_record):
    cursor = dbapi_connection.cursor()
    cursor.execute("PRAGMA foreign_keys=ON")
    cursor.execute("PRAGMA journal_mode=WAL")
    cursor.execute("PRAGMA synchronous=NORMAL")
    cursor.close()
```

**Результат:** FK constraints работают, WAL mode включён

#### Rate limiting Bitrix24 API

**Проблемы:**
1. `KeyError: 'error'` — неправильное имя ключа в статистике
2. `Too many requests` — Bitrix24 API rate limiting (3000 лидов)
3. `Query too old` — callback.answer() вызван слишком поздно
4. **Проверка по адресу** — слишком много запросов, редко находит дубли

**Файлы:**
- `src/bitrix24/duplicates.py` — отключена проверка по адресу, улучшена разбивка
- `config/config.yaml` — увеличен таймаут

**Исправления:**
- `stats[result_type]` → явный маппинг (`duplicates`, `unique`, `errors`)
- `max_parallel`: 5 → **2** (снижение нагрузки на API)
- `rate_limit_delay`: 0.2 → **1.0 сек** (увеличение задержки)
- `request_timeout`: 30 → **60 сек** (для медленных запросов)
- **Отключена проверка по адресу** (find_leads_by_address) — 3-й запрос на каждый лид
- **Автоматическая разбивка на части**:
  - **>1000 лидов**: части по 500 шт. + батчи по 100 шт.
  - **500-1000 лидов**: батчи по 100 шт.
  - **<500 лидов**: обычная обработка
- **Паузы**:
  - Между батчами: 2-5 сек
  - Между частями: 10 сек
- **Прогресс бар**: логирование каждые 100 лидов

**Результат:**
- ✅ KeyError больше не возникает
- ✅ Too many requests реже (2 параллельных запроса вместо 5)
- ✅ Успешная проверка 3000 лидов без ошибок
- ✅ Прогресс виден в логах (📊 Прогресс: 100/3000)
- ✅ Меньше запросов к API (нет проверки по адресу)

**Валидация Bitrix24 Webhook URL**

**Файл:** `src/bitrix24/client.py`

**Добавлено:**
- Проверка HTTPS протокола
- Проверка домена Bitrix24 (.bitrix24.ru, .com, .eu, .by, .kz, .ua, .ge, .am, .md)
- Проверка структуры пути (/rest/{user_id}/{webhook_hash}/)

**Результат:** Защита от подмены webhook URL

**Exponential Backoff для API**

**Файл:** `src/bitrix24/client.py`

**Было:**
```python
await asyncio.sleep(self.retry_delay * (attempt + 1))  # Линейная задержка
```

**Стало:**
```python
delay = (2 ** (attempt + 1)) + random.uniform(0, 1)  # Exponential + jitter
await asyncio.sleep(delay)
```

**Результат:** Надёжная работа при временных проблемах сети
