## 🎯 Ключевые изменения

### 1. Система "Прочее" для сегментов и городов

**Проблема:**
- Менеджеры загружали лиды маленькими порциями (1-5 лидов)
- В сегментах оставались "хвосты" по 1-10 лидов
- Список сегментов засорялся малыми количествами

**Решение:**
- **Сегменты < 10 лидов** → перемещаются в "📦 Прочее (Обыч.)" или "📦 Прочее (Плюсовики)"
- **Города < 10 лидов** внутри сегмента → объединяются в "📦 Прочие (Обыч.)" или "📦 Прочие (Плюсовики)"
- **Разделение по UTC**: Обыч. (< +3ч от МСК), Плюсовики (>= +3ч от МСК)

**Визуализация:**
```
Было:
🏗 Строительство (Москва) — 150 лидов
🏗 Строительство (Казань) — 3 лида      ← "хвост"
🏗 Строительство (Саратов) — 5 лидов   ← "хвост"

Стало:
🏗 Строительство (Москва) — 150 лидов
📦 Прочее (Обыч.)                        ← объединённые "хвосты"
📦 Прочее (Плюсовики)
```

---

### 2. Управление городами с UTC

**Новый функционал:**
- **Таблица `cities`** — хранение городов с UTC offset от Москвы
- **Таблица `pending_cities`** — новые города из CSV ожидают ввода UTC
- **Админ-панель** — "🌍 Управление городами" для добавления UTC
- **Миграция v8** — автоматическое добавление существующих городов с `utc_offset=0`

**Сценарий:**
```
1. Импорт CSV с новым городом "Екатеринбург"
   ↓
2. Город попадает в pending_cities
   ↓
3. Админ заходит в "🌍 Управление городами"
   ↓
4. Вводит: "+2 Екатеринбург"
   ↓
5. Город добавлен в cities, лиды активированы
```

---

### 3. Улучшения для менеджеров

**Лимит 10 лидов не применяется к "Прочее":**
- Менеджер может взять **любое количество** лидов из "📦 Прочее"
- Если в "Прочее" < 10 лидов → бот предлагает взять всё доступное

**Визуализация:**
```
Менеджер выбирает: 📦 Прочее (Обыч.)
   ↓
Бот: "📊 Доступно лидов: 3"
   ↓
Менеджер вводит: "3"
   ↓
Бот: "✅ В категории 'Прочее' доступно только 3 лидов.
      Хотите взять всё доступное количество (3)?"
      
      [✅ Да, взять 3]  [❌ Нет, отменить]
```

## 🔧 Технические детали

### Новые таблицы БД
```sql
CREATE TABLE cities (
    id INTEGER PRIMARY KEY,
    name TEXT UNIQUE NOT NULL,
    utc_offset INTEGER NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE pending_cities (
    id INTEGER PRIMARY KEY,
    name TEXT UNIQUE NOT NULL,
    admin_telegram_id TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### Новые функции CRUD
- `count_other_leads(session, other_type, segment=None)` — подсчёт лидов в "Прочее"

- `admin_pending_cities.py` — управление pending городами
- `handle_city_select()` в `manager_leads.py` — поддержка "Прочие" городов

## 🎯 Ключевые изменения

### 1. Улучшения UX при загрузке лидов

**Что было:**
- После нажатия "✅ Подтвердить" ничего не происходило до завершения загрузки
- Кнопку можно было нажать многократно → ошибки
- Нет задержек между импортами → "Too many requests" от Bitrix24 API

**Что стало:**
- ✅ Сообщение о начале загрузки: "⏳ Загрузка лидов в Bitrix24..."
- ✅ Кнопка удаляется после нажатия → защита от повторных нажатий
- ✅ Задержки 50%: 0.25с между лидами, 1.5с каждые 10 лидов
- ✅ Обработка ошибок callback → если callback истёк, отправка новым сообщением

**Визуализация:**
```
Пользователь нажимает "✅ Подтвердить"
↓
[Сообщение с кнопкой удаляется]
↓
⏳ Загрузка лидов в Bitrix24...

Это может занять несколько минут.
↓
[Идёт загрузка с задержками...]
↓
✅ Выдано 50 лидов
```

- ✅ Нет таймаутов callback
