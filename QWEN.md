# Проект: Telegram-бот для раздачи холодных лидов с интеграцией в Bitrix24

## Обзор проекта

Система представляет собой Telegram-бота для управления холодными лидами с полной интеграцией в Bitrix24 CRM. Бот предназначен для команды из ~20 менеджеров по продажам и включает административный интерфейс для управления лидами, сегментами и аналитикой.

### Основное назначение

- **Импорт лидов**: Загрузка CSV-файлов с холодными лидами в локальную базу SQLite
- **Проверка дублей**: Автоматическая верификация лидов через Bitrix24 REST API (методы `crm.duplicate.findbycomm` и аналоги)
- **Распределение лидов**: Выдача менеджерам до 200 лидов за запрос с учётом сегментов и городов
- **Динамические интерфейсы**: Формирование кнопок только по доступным (не замороженным) сегментам/городам
- **Аналитика**: Метрики по дням/неделям/месяцам (загружено, дубли, выдано, импортировано)
- **Администрирование**: Управление сегментами (заморозка/разморозка), заявками менеджеров, статистикой, экспорт отчётов

### Ключевые особенности

- **Один бот для всех ролей**: Менеджеры и админы работают в одном Telegram-боте с разграничением прав
- **Динамические клавиатуры**: Кнопки формируются на основе реальных данных в БД (только уникальные и активные сегменты/города)
- **Защита от дублей**: Лиды со статусом `DUPLICATE` никогда не выдаются менеджерам
- **Заморозка сегментов**: Логическая приостановка выдачи без физического удаления из БД
- **Работа в РФ**: Архитектура учитывает возможную нестабильность Telegram API (обработка таймаутов, логирование ошибок, восстановление состояния)
- **Таблица сегментов**: Отдельная таблица `segments` для стабильного порядка и управления
- **Пагинация**: Списки сегментов с пагинацией (20 на странице)
- **Доп. поля**: Импорт в Bitrix24 с пользовательскими полями (Тип услуги, Источник телефона)
- **Статистика менеджера**: Детализация по периодам (день/неделя/месяц), выдано/импортировано
- **Статистика по менеджерам**: Сводка по всем менеджерам с детализацией (админ)
- **Экспорт CSV**: Статистика менеджера и отчёты по менеджерам

## Building and Running

### Предварительные требования

- Python 3.10+
- Telegram Bot Token (получить у @BotFather)
- Bitrix24 Incoming Webhook (с правами CRM и Users)
- SQLite 3 (встроен в Python)

### Установка зависимостей

```bash
pip install -r requirements.txt
```

### Конфигурация

Создайте файл `.env` в корне проекта:

```env
# Telegram Bot
TELEGRAM_BOT_TOKEN=1234567890:ABCdefGHIjklMNOpqrsTUVwxyz

# Bitrix24 Webhook
BITRIX24_WEBHOOK_URL=https://your-portal.bitrix24.ru/rest/1/your-webhook/
BITRIX24_WEBHOOK_SECRET=your-webhook-secret

# Database
DATABASE_PATH=./data/leads.db

# Uploads Folder
UPLOADS_FOLDER=./uploads

# Admin Telegram IDs (список админов)
ADMIN_TELEGRAM_IDS=123456789,987654321

# Logging
LOG_LEVEL=INFO
LOG_FILE=./logs/bot.log

# Proxy (опционально, для работы в РФ)
TELEGRAM_PROXY_URL=http://proxy-server:port
```

### Запуск бота

```bash
# Разработка
python src/bot/main.py

# Production (с логированием)
PYTHONPATH=. python src/bot/main.py --prod

# Docker
docker-compose up -d
```

### Запуск тестов

```bash
# Все тесты
pytest tests/ -v

# С покрытием
pytest tests/ --cov=src --cov-report=html

# Отдельные модули
pytest tests/test_import.py -v
pytest tests/test_duplicates.py -v
pytest tests/test_segment_extraction.py -v
```

### Миграция сегментов

Для обновления существующих лидов и извлечения сегментов из поля `phone_source`:

```bash
# Windows
migrate_segments.bat

# Linux/Mac
python -m src.database.migrate_segments
```

**Что делает миграция**:
1. Находит все лиды с заполненным `phone_source`
2. Извлекает сегмент из формата `"сегмент!остальные данные"`
3. Обновляет поле `segment` в БД
4. Логирует процесс

**Важно**: Перед миграцией рекомендуется сделать резервную копию БД.

#### Поле `phone_source`

**Формат**: `"название сегмента!остальные данные"`

**Примеры**:
- `"быстровозводимые здания!СБП_КраснЯр_Казань__2026-03-18_14_17_02.json"`
- `"Автосалоны!Тестовые данные"`
- `"Грузовые автомобили!СПб_2026-03-18"`

**Логика извлечения сегмента**:
1. Если есть `"!"` — берётся часть до него (например, `"быстровозводимые здания"`)
2. Если нет `"!"` — берётся первое слово по пробелу
3. Если пусто — возвращается `"Без сегмента"` или fallback из поля `Название лида`

**Миграция данных**: Для обновления существующих лидов используется скрипт `migrate_segments.bat`, который извлекает сегменты из `phone_source` и обновляет поле `segment`.

## Статусы лидов и поток данных

### Жизненный цикл лида

```
NEW → (проверка дублей) → UNIQUE/DUPLICATE
UNIQUE → (запрос менеджером) → ASSIGNED → (импорт в Bitrix24) → IMPORTED
UNIQUE → (ошибка импорта) → ERROR_IMPORT
```

### Статусы

| Статус | Описание | Может быть выдан |
|--------|----------|------------------|
| `NEW` | Только что импортирован, ещё не проверен | ❌ Нет |
| `UNIQUE` | Проверен, дублей не найдено, доступен | ✅ Да (если сегмент не заморожен) |
| `DUPLICATE` | Найден дубль в Bitrix24 | ❌ Нет |
| `ASSIGNED` | Выдан менеджеру (в процессе импорта) | ❌ Нет |
| `IMPORTED` | Успешно создан в Bitrix24 | ❌ Нет |
| `ERROR_IMPORT` | Ошибка при импорте в Bitrix24 | ⚠️ Да (после повторной попытки) |

### Поток данных

1. **Импорт CSV**: Админ загружает CSV → лиды создаются со статусом `NEW`
2. **Проверка дублей**: Автоматически после импорта → статус меняется на `UNIQUE` или `DUPLICATE`
3. **Запрос менеджера**: Менеджер выбирает сегмент/город/количество → бот проверяет доступность (с учётом заморозки) → при нехватке предлагает меньшее количество → подтверждение → статус `ASSIGNED`
4. **Импорт в Bitrix24**: Создание лида через REST API → статус `IMPORTED` или `ERROR_IMPORT`
5. **Логирование**: Все этапы записываются в таблицу `logs`
6. **Аналитика**: Агрегация метрик по дням/неделям/месяцам
7. **Очистка**: Ежемесячная очистка логов и старых лидов (`DUPLICATE`, `IMPORTED`)

## Сценарии Telegram-бота

### Роль менеджера

#### Регистрация (первый запуск)

1. Менеджер нажимает `/start`
2. Бот просит ввести Имя и Фамилию (свободным текстом)
3. Создаётся заявка со статусом `PENDING_APPROVAL`
4. Админ получает уведомление с заявкой
5. Админ выбирает или вводит ID пользователя Bitrix24 → подтверждает привязку
6. Менеджер получает уведомление об активации
7. Статус меняется на `ACTIVE` → доступен полный функционал

#### Главное меню менеджера

```
📋 Получить лиды
📊 Моя статистика
ℹ️ О боте
```

#### Сценарий получения лидов

1. **Выбор сегмента** (inline-клавиатура):
   ```
   🏗 Строительство
   🏭 Производство
   🛒 Розница
   💼 Услуги
   ⬅️ Назад
   ```
   *Отображаются только сегменты с доступными (`UNIQUE`) и не замороженными лидами*

2. **Выбор города** (inline-клавиатура):
   ```
   Москва
   Санкт-Петербург
   Казань
   ⬅️ Назад
   ```
   *Отображаются только города с доступными лидами в выбранном сегменте*

3. **Запрос количества**:
   ```
   Введите количество лидов (максимум 200):
   ```

4. **Проверка доступности**:
   - Если запрошено 200, а доступно 10:
     ```
     ⚠️ В выбранном сегменте/городе доступно только 10 лидов вместо 200.
     Выдать доступное количество (10)?

     ✅ Да, выдать 10
     ❌ Нет, отменить
     ```

5. **Подтверждение и выдача**:
   ```
   ✅ Выдано 10 лидов по сегменту "Строительство" (Москва)
   Лиды импортированы в Bitrix24.
   ```

#### Моя статистика

```
📊 Ваша статистика за:

📅 Сегодня: 50 лидов
📅 Эта неделя: 200 лидов
📅 Этот месяц: 800 лидов

⬅️ Назад
```


1. Админ выбирает «⚙️ Управление сегментами»

   ```

   ```

## Интеграция с Bitrix24

### Входящий вебхук

Все обращения к Bitrix24 осуществляются через **входящий вебхук** с минимально необходимыми правами:

- **CRM**: `crm.lead.*`, `crm.duplicate.*`, `crm.contact.*`, `crm.company.*`
- **Users**: `user.*`

### Конфигурация вебхука

```env
BITRIX24_WEBHOOK_URL=https://your-portal.bitrix24.ru/rest/1/your-webhook/
```

### Используемые методы API

| Метод | Назначение | Права |
|-------|------------|-------|
| `crm.duplicate.findbycomm` | Поиск дублей по коммуникациям (телефон) | `crm` |
| `crm.lead.add` | Создание лида | `crm` |
| `crm.lead.update` | Обновление лида | `crm` |
| `crm.lead.get` | Получение данных лида | `crm` |
| `user.get` | Получение данных пользователя | `user` |

### Проверка дублей

**Поля для поиска**:

- Телефон (основной критерий)
- Название компании (дополнительно)
- Адрес (дополнительно)

**Алгоритм**:

1. После импорта CSV лиды со статусом `NEW` попадают в очередь проверки
2. Для каждого лида вызывается `crm.duplicate.findbycomm` с телефоном
3. Если найден дубль → статус `DUPLICATE`
4. Если дублей нет → статус `UNIQUE`
5. Дополнительно проверяется название компании и адрес (независимо от телефона)
6. Результаты записываются в `logs`

### Импорт лидов

**Алгоритм**:

1. Менеджер подтверждает получение N лидов
2. Для каждого лида:
   - Статус меняется на `ASSIGNED`
   - Вызывается `crm.lead.add` с полями из CSV
   - Передаются пользовательские поля:
     - `UF_CRM_1745433492760` (Тип услуги)
     - `UF_CRM_1768999878619` (Источник телефона)
   - При успехе → статус `IMPORTED`, сохраняется `bitrix24_lead_id`
   - При ошибке → статус `ERROR_IMPORT`, логируется ошибка
3. Менеджер получает уведомление о результате

## Безопасность

### Хранение секретов

- **Telegram Bot Token**: Переменная окружения `TELEGRAM_BOT_TOKEN` или файл `.env`
- **Bitrix24 Webhook URL**: Переменная окружения `BITRIX24_WEBHOOK_URL`
- **Webhook Secret**: Переменная окружения `BITRIX24_WEBHOOK_SECRET`
- **Файл `.env`**: Должен быть исключён из Git (`.gitignore`)

### Проверка ролей

- Все обработчики команд проверяют роль пользователя в БД
- Админские команды доступны только пользователям с `role = 'admin'`
- Telegram ID админов конфигурируются в `.env` (`ADMIN_TELEGRAM_IDS`)

### Защита данных

- **SQLite**: Файл БД хранится в защищённой директории (доступ только приложению)
- **CSV**: Файлы в `/uploads` доступны только приложению
- **Логи**: Логи записываются в защищённую директорию `/logs`

### Рекомендации по развёртыванию

- Использовать HTTPS для всех внешних соединений
- Ограничить доступ к серверу по IP (если возможно)
- Регулярно обновлять зависимости
- Включить логирование всех ошибок и подозрительных действий
- Рассмотреть использование прокси/VPN для стабильной работы Telegram в РФ

### Особенности работы в РФ

- **Обработка таймаутов**: Все вызовы Telegram API и Bitrix24 API должны иметь таймауты и повторные попытки
- **Логирование сбоев**: При недоступности Telegram логируются ошибки, но данные в БД сохраняются
- **Восстановление состояния**: После перезапуска бот должен корректно восстанавливать состояние из БД
- **Прокси**: На уровне инфраструктуры (сервер/сеть) может потребоваться настройка прокси/VPN
